    ```python
    """Spike runner — safely run external graph-analysis tools on isolated workspace copies."""
    from __future__ import annotations

    import json
    import shutil
    import subprocess
    import tempfile
    from datetime import datetime, timezone
    from pathlib import Path
    from dataclasses import dataclass, field


    KNOWN_SPIKES: dict[str, SpikeDefinition] = {
        "graphify": SpikeDefinition(
            name="graphify",
            description="Graphify-style ingestion analysis — extract candidate tags and keywords from refs",
            command_template="{tool_path} {workspace_copy}",
            expected_output_paths=["candidates.json", "tags.json"],
        ),
        "infranodus": SpikeDefinition(
            name="infranodus",
            description="InfraNodus-style graph exploration — analyze graph structure for insight patterns",
            command_template="{tool_path} --input {workspace_copy}/connections.json",
            expected_output_paths=["report.json"],
        ),
    }
    ```

    Classes and functions:

    1. **SpikeDefinition** dataclass:
       - `name: str` — spike tool name
       - `description: str` — what this spike evaluates
       - `command_template: str` — shell command with {workspace_copy} and {tool_path} placeholders
       - `expected_output_paths: list[str]` — relative output file paths to capture
       - `requires_deps: bool = True` — whether the temp copy needs connections.json and cards/

    2. **SpikeResult** dataclass:
       - `success: bool`
       - `tool_name: str`
       - `started_at: str`
       - `duration_seconds: float`
       - `workspace_path: str` — original (not temp) workspace path
       - `temp_workspace_path: str` — the temp copy path
       - `outputs: dict[str, str]` — captured output files: filename → content
       - `stdout: str` — command stdout
       - `stderr: str` — command stderr
       - `error: str | None` — error message if failed

    3. **run_spike(tool_name, workspace, tool_path=None, install_root=None) -> SpikeResult:**
       - Validate tool_name is in KNOWN_SPIKES
       - Create temp dir via `tempfile.mkdtemp(prefix="construct-spike-")`
       - Copy workspace to temp dir:
         - If `requires_deps`: copy `cards/`, `connections.json`, `domains.yaml`, `refs/`
         - Use `shutil.copytree()` or `shutil.copy2()` for individual items
         - NOTE: Skip large dirs like `views/build/` and `digests/`
       - Resolve tool_path (default to `tool_name` in PATH)
       - Format command string with paths
       - Run via `subprocess.run()` with timeout=300 (5 min)
       - Capture stdout, stderr, return code
       - Collect outputs from `expected_output_paths`
       - Write results to `install_root/log/spike-results/{tool_name}-{timestamp}.json`
       - Clean up temp dir
       - Return SpikeResult

    4. **list_spikes() -> list[dict]:**
       - Return list of KNOWN_SPIKES definitions (name, description)

    5. **register_spike_commands(app: typer.Typer) -> None:**
       - Add `spike` command group to a typer app

    Error handling:
    - If tool not found in PATH: return SpikeResult with `success=False, error="Tool '{tool_name}' not found in PATH"`
    - If timeout: return SpikeResult with `success=False, error="Spike timed out after 300s"`
    - If temp copy fails (workspace doesn't exist): return SpikeResult with `success=False, error=f"Workspace not found: {workspace}"`
    - Always clean up temp dir, even on failure (use try/finally)

    Follow existing codebase conventions: `snake_case` functions, dataclass for result types, `from __future__ import annotations`, typed signatures.

    # Skill: Spike Run

    **Trigger:** User wants to evaluate an external graph-analysis tool without risking canonical workspace data.
    **Agent:** CONSTRUCT (orchestrator)
    **Produces:** `log/spike-results/{tool}-{timestamp}.json` with captured outputs.

    ---

    ## Procedure

    ### Step 0: Prerequisites
    - The external tool must be installed and available in PATH
    - Workspace must exist with cards/ and connections.json

    ### Step 1: List available spike types
    `construct spike list`

    ### Step 2: Run the spike
    `construct spike run <tool-name> --workspace <path> [--tool-path <path>]`

    ### Step 3: Review results
    Results are at `log/spike-results/{tool}-{timestamp}.json`

    ## Spike Types

    ### Graphify: Tag/Keyword Extraction Analysis (SPK-02)

    **Purpose:** Evaluate whether Graphify-style ingestion analysis produces useful candidate tags and keywords from ref source material.

    **How to run:**
    1. `construct spike run graphify --workspace <path>`
    2. Read output candidates
    3. Compare against existing `search-seeds.json`
    4. Evaluate: Are candidates novel? Relevant? Worth curating?

    **What to evaluate:**
    - Quality of extracted tags (precision, recall)
    - Novelty vs existing search-seeds patterns
    - Potential for automated candidate generation during ingestion

    **Interpreting results:**
    - Strong candidates → route through tag pipeline (Phase 6 tag-extraction pipeline)
    - Marginal candidates → note as future tuning opportunities
    - No useful output → tool may not fit the workspace structure

    **Output:** This informs SPK-02/03 decisions about tag pipeline design.

    ### InfraNodus: Graph Exploration Analysis (SPK-04)

    **Purpose:** Evaluate whether InfraNodus-style graph-guided exploration reveals insight patterns that could inform future UI and workflow design. Per D-09: this is evaluated via the spike framework, NOT built as a permanent feature.

    **How to run:**
    1. `construct spike run infranodus --workspace <path> --tool-path <path-to-infranodus>`
    2. Read graph structure report
    3. Identify structural patterns: clusters, bottlenecks, bridging nodes

    **What to evaluate:**
    - Which graph metrics are most useful for insight discovery
    - UI patterns that could expose these insights
    - Integration points with existing bridge detection pipeline

    **Interpreting results:**
    - Useful patterns → document as UI spec input for v0.4
    - Redundant with existing bridge detection → note as lower priority
    - Requires too much external tooling → document as deferred

    **Output:** Design input for v0.4 UI patterns — NOT workspace modifications.
