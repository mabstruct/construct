    2. `async def handle_call_tool(params: types.CallToolRequestParams) -> types.CallToolResult`:
       - Get capability from registry: `cap = get_registry().get_by_mcp_name(params.name)`
       - Convert `params.arguments` (dict) to handler kwargs. Use `cap.input_model.model_validate(params.arguments)` for validation.
       - Call handler: Since existing handlers are synchronous, use `await anyio.to_thread.run_sync(cap.handler, **validated_args.model_dump())`
         Actually, handlers take structured args, not raw kwargs. The pattern from catalog.py is handler(path), handler(workspace, card_data, ...), etc.
         For now, call handlers with the validated input model's dump:
         ```python
         try:
             validated = cap.input_model.model_validate(params.arguments or {})
             # For workspace.validate: handler takes a Path
             # For knowledge ops: handler takes workspace_root + kwargs
             # The handler signature depends on the capability.
             # Use validated.model_dump() and pass as **kwargs for now
             result = await anyio.to_thread.run_sync(
                 _call_handler_sync, cap, validated
             )
         ```
       
       Better approach — separate dispatch per capability pattern:
       ```python
       async def handle_call_tool(params):
           try:
               cap = get_registry().get_by_mcp_name(params.name)
               args = params.arguments or {}
               # Run handler in thread pool (all handlers are sync)
               result = await anyio.to_thread.run_sync(cap.handler, **args)
               # result is OperationResult or ValidationReport
               return types.CallToolResult(content=[
                   types.TextContent(type="text", text=json.dumps(_serialize_result(result), indent=2))
               ])
           except Exception as exc:
               return types.CallToolResult(
                   content=[types.TextContent(type="text", text=json.dumps({"error": str(exc)}))],
                   isError=True,
               )
       ```

       Use `_serialize_result(result)` helper:
       ```python
       def _serialize_result(result):
           if hasattr(result, "model_dump"):
               return result.model_dump(mode="json")
           if hasattr(result, "__dataclass_fields__"):
               return {f: getattr(result, f) for f in result.__dataclass_fields__}
           if isinstance(result, (list, tuple)):
               return [str(item) for item in result]
           return str(result)
       ```

    3. `def create_server() -> Server`:

    4. `def run_server() -> None`:
       ```python
       def run_server() -> None:
           """Entry point for `construct mcp` — runs until stdin closes."""
           app = create_server()
           
           async def arun():
               async with stdio_server() as (read_stream, write_stream):
                   await app.run(read_stream, write_stream, app.create_initialization_options())
           
           anyio.run(arun)
       ```

    Note: The `mcp` library's `Server` uses `on_list_tools`/`on_call_tool` as handler attributes. The v1.x API sets these as properties on the Server instance:
    ```python
    app = Server("construct")
    app.on_list_tools = handle_list_tools
    app.on_call_tool = handle_call_tool
    ```
    If the library uses constructor kwargs instead:
    ```python
    app = Server("construct", on_list_tools=handle_list_tools, on_call_tool=handle_call_tool)
    ```
    Use whichever pattern works — try constructor kwargs first, fall back to attribute assignment.

    The `run_server()` MUST use `anyio.run(arun)` not `asyncio.run()` because `stdio_server()` is an anyio context manager.

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| MCP stdio transport (Claude/Cursor → MCP server) | stdin/stdout JSON-RPC messages between client and server within the same process |
| MCP tool dispatch (tool name → handler) | Tool call requests cross from MCP protocol into capability handlers |
| Workspace file access (handlers → filesystem) | Capability handlers read/write workspace files |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-03-09 | T (Tampering) | MCP stdio transport JSON parsing | mitigate | `mcp>=1.0` library handles JSON-RPC framing. Arguments validated through `cap.input_model.model_validate()` before dispatch. |
| T-03-10 | T (Tampering) | handle_call_tool — unknown tool name | mitigate | `get_by_mcp_name()` raises `KeyError` for unknown tools. Caught in try/except, returns `isError=True` response. |
| T-03-11 | E (Elevation) | Thread pool handler dispatch (anyio.to_thread.run_sync) | accept | Handlers are trusted callables from the capability registry. All write operations go through pre-write validation gates established in Phase 1. |
| T-03-12 | S (Spoofing) | MCP client identity | accept | Local-only architecture — stdio transport bound to parent process. No remote clients can connect. |
| T-03-13 | I (Info Disclosure) | MCP tool results serialization | accept | Results serialized as JSON. Same information accessible via CLI. Local filesystem only. |
</threat_model>
