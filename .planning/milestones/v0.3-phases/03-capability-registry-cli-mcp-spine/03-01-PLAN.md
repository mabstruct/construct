       - `id: str` — unique capability ID, e.g. "workspace.validate"
       - `name: str` — human-readable name
       - `description: str`
       - `input_model: InputModel` — Pydantic model class (not instance) for input validation & JSON Schema generation
       - `output_model: OutputModel` — Pydantic model class for output schema
       - `handler: Callable[..., Any]` — direct callable reference

       - `mcp_tool_name: Optional[str] = None` — MCP tool name if surfaced via MCP

       - `_capabilities: dict[str, CapabilityRecord]` — internal store
       - `register(record: CapabilityRecord) -> None` — adds to dict, raises `ValueError` on duplicate `id` (message: f"Capability '{record.id}' is already registered")
       - `get(cap_id: str) -> CapabilityRecord` — returns record, raises `KeyError` with message including available IDs
       - `list() -> list[CapabilityRecord]` — returns all capabilities sorted by ID
       - `list_by_cli(cli_name: str) -> list[CapabilityRecord]` — returns capabilities with matching cli_name (for group-level lookups)
       - `get_by_mcp_name(name: str) -> CapabilityRecord` — returns record matching mcp_tool_name, raises `KeyError`
       - `list_mcp_tools() -> list[dict]` — returns list of dicts with keys: `name` (mcp_tool_name), `description`, `input_schema` (from `input_model.model_json_schema()`). Only includes capabilities where mcp_tool_name is set.
       - `__len__() -> int` — returns count

    5. All handler references are stored as-is (no module-path strings). Per D-03: "Calling a capability is cap.handler(**inputs)."

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Registry API (Python → registry) | Code calling registry.get() or registry.register() within the Python process |
| Handler dispatch (catalog → external services) | Handlers called via cap.handler(**inputs) may access filesystem |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-03-01 | S (Spoofing) | catalog.py handler references | accept | Handler references are module-level callables defined in trusted library code — not user-injectable. No authentication boundary exists in local-first architecture. |
| T-03-02 | T (Tampering) | registry.py register() | mitigate | `register()` raises `ValueError` on duplicate ID, preventing unvalidated overwrite. Registry created once via `create_registry()` and cached. |
| T-03-03 | I (Info Disclosure) | list_mcp_tools() JSON Schema | accept | JSON Schema exposes model field names and types — same information available in source code. Local filesystem only. |
| T-03-04 | E (Elevation) | handler dispatch | mitigate | `cap.handler(**inputs)` dispatches to trusted callables only. Handler references are set at registration time from trusted module imports, not user input. |
</threat_model>
